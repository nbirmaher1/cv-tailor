# Tailor CV to a job posting

Inputs: a CV file path (PDF or .docx), a job description (a URL, or pasted text if no URL works), optionally a photo file to include, optionally free-text notes/comments from the candidate (e.g. contact/location updates, visa/sponsorship status, or extra detail about a role that isn't in the CV text), and optionally an output format (`pdf` or `docx`; default `pdf`) and/or explicit output paths. When invoked directly by a user (not the web app), ask for whatever of these wasn't given (notes are optional — don't block on them).

Project root: `~/projects/cv-tailor`. Python venv (has `playwright`, `python-docx`, `pypdf`): `~/projects/cv-tailor/venv/bin/python3`.

## Steps

1. **Read the CV.**
   - If the file is a `.pdf`, read it directly with the Read tool (it supports PDF).
   - If the file is a `.docx`, extract its text with:
     `~/projects/cv-tailor/venv/bin/python3 ~/projects/cv-tailor/scripts/extract_docx.py <path>`

2. **Get the job description.**
   - If given a URL, try WebFetch on it first.
   - LinkedIn and some ATS pages often block fetching or return a login wall / near-empty content. If the fetch fails, returns very little text, or looks like a login page, tell the user and ask them to paste the job description text instead. Don't guess at job content.
   - If given raw pasted text already, use it directly.

3. **Extract structure from the CV into a canonical JSON content record** (this is the single source of truth used for either output format later), preserving the CV's actual section structure rather than collapsing everything into a fixed set of buckets:

   ```json
   {
     "full_name": "", "target_title": "", "email": "", "phone": "", "location": "", "links": "",
     "work_authorization": "",
     "photo_path": null,
     "summary": "",
     "experience": [{"title": "", "company": "", "location": "", "dates": "", "bullets": [""]}],
     "education": [{"degree": "", "school": "", "location": "", "dates": ""}],
     "skills": [{"category": null, "items": [""]}],
     "languages": [""],
     "extra_sections": [{"heading": "", "items": [""]}]
   }
   ```

   - `work_authorization` is a short line (e.g. "EU citizen — no visa sponsorship needed") that renders inline with contact info (email/phone/location/links), not as its own section. Leave it `""` unless the source CV or candidate notes actually say something about work authorization/visa/sponsorship.

   - `skills` may have multiple groups if the source CV labels subcategories (e.g. "Analysis & Visualization" vs "Big Data & Engineering"); use a single group with `category: null` if it doesn't.
   - **If the source CV has Languages as its own distinct section or clearly-separate subsection, put it in `languages`, not folded into `skills`.** Leave `languages` empty if the source CV has none.
   - Any other distinct section in the source CV (certifications, projects, publications, etc.) goes in `extra_sections` as its own entry — never merged into the nearest similar-looking bucket.
   - If a photo file was provided as input, set `photo_path` to its absolute path; otherwise leave it `null`.
   - **If candidate notes/comments were provided, apply them to this JSON now, before job-tailoring in step 4.** Update fields they explicitly override (e.g. `location` for a relocation, `phone`/`email` for changed contact info, `work_authorization` for anything about visa/sponsorship/right-to-work). Fold anything else into the most relevant existing field or bullet; only fall back to a new `extra_sections` entry if it genuinely doesn't fit any existing field (contact-line details like work authorization never need their own section). Treat these notes as first-person authoritative statements from the candidate, not claims to independently verify against the source CV — they're additions/corrections from the person the CV is about, not fabrication. They must actually show up in the rendered output, not just be read as background context.
   - **If the source document is a long, detailed "master CV"** (many roles, many bullets per role — a full career history rather than a targeted resume), extract it *faithfully and completely* into this JSON regardless of length. Do not pre-filter, trim, or drop anything at this stage — capturing everything that exists in the source is step 3's only job; something left out here can never be recovered later. Selecting what to keep for this specific job happens in step 4.
   - **Normalize `location` to "City, Country" (or "City, State" for the US)** — drop a full street address or postal code if the source CV includes one. This is a formatting normalization, not tailoring, so apply it regardless of the job posting; never invent a location that isn't in the source.

4. **Tailor the content to the job posting** (this JSON is a draft — it gets reviewed in step 5):
   - **Before rewriting anything, extract the JD's 8-10 most critical required skills, tools, and responsibilities.** This is the target list you'll weave into rewritten bullets and the skills section throughout the rest of this step — keywords belong inside the proof (a bullet that demonstrates the skill), not just listed in Skills.
   - Rewrite `summary` as a tight 3-4 sentence professional summary: lead with language that mirrors the JD's target title/domain, name the competencies from the extracted JD list that the CV genuinely supports, and include at least one concrete proof point (a metric, scale, or outcome) drawn from the CV. No generic "results-driven professional" filler — every sentence must earn its place.
   - **Judge relevance of every role and every bullet against this specific job posting before rewriting anything.** For each experience entry, decide: core (directly relevant to this job — keep and lead with it), tangential (some relevance — keep a trimmed version), or irrelevant (no meaningful relevance to this job — cut the whole entry). Do the same at the bullet level within entries you keep.
   - **Select and cut, don't just lightly trim, when the source has more material than a targeted resume needs** (the common case for a long "master CV" source, and a no-op for a source that's already a short polished CV since everything in it will typically already read as relevant). Concretely:
     - Drop whole roles that are irrelevant to this job when there's enough other, stronger material to cover the page — e.g. an unrelated early-career job with no transferable relevance to this posting.
     - **Bullet caps per role: 3-5 bullets for the most recent/most relevant (core) role(s); 1-2 bullets for older or tangentially relevant roles.** Rank by fit and cut down to the cap — don't keep a bullet just because it's impressive if it's not one of the strongest matches for this posting.
     - **Deduplicate before finalizing: no two bullets within the same role should describe the same underlying piece of work.** If the source has separate bullets touching the same project/deliverable from different angles (e.g. one bullet on building a dashboard, another on that same dashboard's design/adoption), merge them into a single, stronger bullet that combines the strongest details from each rather than keeping both — a fragmented achievement split across two bullets reads as padding, not two accomplishments. This applies within a role even when the two source bullets used different wording.
     - Do relevance-based cutting *first*; only tighten wording for space after that still leaves you over target.
   - **Every surviving bullet must read as an achievement, not a duty.** Concretely:
     - Start with a strong action verb (Led, Built, Reduced, Delivered, Negotiated) — never "Responsible for", "Worked on", "Helped with", or other passive/duty framing.
     - Follow an "achieved X by doing Y" shape and preserve every quantifiable metric from the source (%, $, time saved, volume, headcount) exactly as given — never invent or round up a number that isn't in the source.
     - If a bullet has no impact or metric and doesn't directly serve a JD requirement, cut it entirely — a shorter list of strong bullets beats a longer list padded with duty-only lines.
     - Translate technical detail into plain-language business impact where the source actually supports it (e.g. "wrote SQL queries" → "cut reporting turnaround by 40% by rebuilding the [X] queries", only if the CV states that outcome); if the source only states the technical fact with no stated business impact, report the technical fact concisely rather than inventing an impact for it.
     - **Hard length cap: each bullet must render to at most 2 lines in the PDF template (roughly 180-200 characters / 28-32 words).** If a rewritten bullet runs longer, cut secondary/qualifying detail down to the strongest single claim rather than letting it wrap further. If a source bullet is a long narrative paragraph (common in a detailed master CV), condense it to this length rather than carrying its full length over verbatim.
     - **Weave the JD's terminology (from the top 8-10 list above) directly into bullets wherever the CV genuinely demonstrates that skill or responsibility** — this is the primary place keywords should live, not just the skills list. Never adopt a posting's term for a skill/tool/responsibility the CV doesn't actually demonstrate; that's keyword-stuffing and is covered by the no-invention rule below.
   - **Skills section — this is the single fastest place the CV overflows past 1 page, so cut it hard, not lightly:**
     - Target **3-5 categories (or a single flat list if the source isn't categorized), with 3-5 items each — a hard cap of ~20 total skill items** across the entire section, regardless of how large the source skills matrix is.
     - Items are tool/skill names only, at most a short 2-3 word qualifier — never a parenthetical clause explaining scope (that nuance belongs in a bullet, attached to a specific claim, not repeated here).
     - Drop entire categories and individual items with no relevance to this posting, the same way you'd drop an irrelevant role. If cutting a category to size would leave only 1-2 stray items, fold them into a related category instead of keeping a near-empty one.
     - Adjust remaining item wording/order to match the posting's terminology only where it's a genuine, truthful match.
   - **Length target: 1 page by default.** A second page is only justified when the candidate has 5+ years of relevant experience (or genuinely extensive, job-relevant project work) *and* everything on that second page is fully job-relevant, not padding kept because cutting felt harsh — in that case a full, deliberate 2 pages is fine. **Never produce more than 2 pages** (this tool doesn't target academic/medical CVs, where longer is normal). A lightly-spilled ~1.5-page result is not an acceptable resting point either way — it means either cut harder to reach 1 page, or the content genuinely justifies committing to a full 2 pages. See step 8 for how this is enforced against the actual rendered output.
   - Reorder bullets within each kept entry to foreground what's relevant to this job.
   - **Never invent experience, skills, titles, or dates that aren't in the source CV.** Rephrasing, re-emphasizing, and reordering existing content is fine; fabrication is not. If the CV is missing something the job clearly wants, leave it out rather than inventing it.

5. **Review the draft JSON with an independent subagent.** Use the Agent tool (`subagent_type: general-purpose`, run in the foreground — you need its result before continuing) with a fully self-contained prompt that includes: the complete original CV text, the complete job description text, any candidate notes/comments provided, and the tailored draft JSON. Ask it to check, and report back explicitly:
   - **Accuracy**: every claim in the draft (skills, achievements, titles, dates, responsibilities) must be traceable to the original CV or to the candidate's notes. Rephrasing/reordering/re-emphasizing is fine; anything added, exaggerated, or not supported by either source is not — list each such claim. This includes phrasing borrowed from the job posting: if a bullet now uses the posting's terminology, verify the underlying CV content genuinely supports it — flag any phrase that reads as keyword-stuffing or overstates what the original bullet said.
   - **Relevance and selection quality**: does the draft actually address the job posting's key requirements/skills/responsibilities? Flag major requirements the draft ignores when the source CV did have relevant material for them, and flag any rewrites that are generic filler rather than a genuine match. If the source was a long master CV, also flag the reverse problem: low-relevance roles/bullets that were kept or barely trimmed while stronger, more relevant material got cut or under-emphasized instead — that's a selection error, not a completeness issue.
   - **Bullet quality and caps**: flag any bullet that reads as a duty/responsibility line rather than an achievement (passive framing, no impact, no metric where the source had one available), any bullet that would clearly wrap past 2 rendered lines, and any role that exceeds its cap (3-5 bullets for core roles, 1-2 for tangential/older ones). Flag a Skills section that exceeds ~20 total items, has more than 5 categories, or still has parenthetical scope explanations instead of plain skill names. Flag any pair of bullets within the same role that describe the same underlying project/deliverable rather than distinct pieces of work — that's a duplication error and should be merged into one bullet.
   - **Completeness (of what's relevant, not of the whole source)**: contact info and any distinct source sections that are still relevant to this job (languages, certifications, etc.) shouldn't have vanished, and — if notes were provided — everything in them was actually reflected somewhere in the draft. **Do not flag the intentional dropping of low-relevance roles/bullets from a long source CV as a completeness problem — that's expected, correct behavior, not an omission.** Only flag a drop as a real completeness issue if it removed the CV's only evidence of a requirement the job posting explicitly asks for.
   - Have it respond with either `PASS` or a specific, actionable list of fixes (quote the problematic text and say what's wrong).

   If the review returns fixes, revise the JSON accordingly and run the review subagent once more on the revised draft. Cap it at 2 review rounds total — if issues remain after the second round, proceed with the best version and note the unresolved concern in your summary to the user rather than looping indefinitely. Keep track of what was flagged and fixed so you can summarize it in step 9.

6. **Determine the output format** (`pdf` or `docx`, default `pdf` if not specified) and output paths (default `~/projects/cv-tailor/output/<short-job-slug>.<ext>` unless explicit paths were given as input).

7. **Render:**
   - **PDF path**: read `~/projects/cv-tailor/templates/default.html` (unless the user pointed you at a different style template) and produce a filled copy with the reviewed JSON content substituted for the `{{PLACEHOLDER}}` tokens — repeat the example entry blocks (Experience, Education, Skills) per item, most-recent-first for experience/education. If `photo_path` is set, keep the `.header`/`.photo` markup; if not, drop the photo `<img>` and header wrapper entirely (don't leave a broken image). Drop any empty contact-line span (phone/links/`work_authorization` are all optional) rather than leaving a stray separator. **Don't modify the template to introduce multi-column layouts, tables, icons, or graphical skill-rating bars** — the single-column layout with standard section headers (Summary/Experience/Education/Skills) is deliberate for ATS parsing reliability. Write the filled HTML to `<output>.html`, then render:
     `~/projects/cv-tailor/venv/bin/python3 ~/projects/cv-tailor/scripts/render_pdf.py <output>.html <output>.pdf`
     `render_pdf.py` prints the resulting page count after rendering (e.g. `Wrote <output>.pdf (2 pages)`) — note it, you'll use it in step 8.
   - **DOCX path**: write the reviewed JSON content record to `<output>.json` (matching the schema in step 3 exactly, `photo_path` included if set), then render:
     `~/projects/cv-tailor/venv/bin/python3 ~/projects/cv-tailor/scripts/render_docx.py <output>.json <output>.docx`

8. **Visually inspect the rendered output before presenting it, and enforce the step-4 length policy against the actual rendered result.**
   - For PDF: use the Read tool on the output PDF and actually look at the rendered page(s) — don't just trust that the HTML was well-formed. Check for uneven spacing around separators/bullets, overlapping or cut-off text, awkward gaps from empty fields, content spilling onto a near-empty extra page, or anything that would look unpolished on a real job application. If you find a template-level bug, fix `templates/default.html` too so future runs don't repeat it.
     - **Use `render_pdf.py`'s reported page count as a strict gate, not a soft signal.** 1 page is the target and the expected outcome for most candidates. If the result is 2 pages, apply the step-4 test: does the candidate have 5+ years of relevant experience or genuinely extensive relevant project work, AND is every line on the second page job-relevant rather than filler kept out of reluctance to cut? If both hold, a full 2 pages is an acceptable, deliberate outcome. If either doesn't hold, go back to step 4 and cut further. **A lightly-filled second page (~1.5 pages) is never an acceptable stopping point** — either cut down to a clean 1 page, or the content genuinely earns a full 2 pages. **3+ pages is never acceptable** regardless of experience level — go back to step 4 and cut harder.
     - **When cutting for length, cut in this order: (1) Skills section down to the step-4 cap (~20 items, 3-5 categories), (2) tangential/older-role bullets down to their 1-2 cap, (3) core-role bullets down to their 3-5 cap.** Skills is evidence-free listing rather than proof of the work you actually did, so it's the fastest way to reclaim space at the lowest cost to the application. Re-render and recheck after each round of cuts.
   - For DOCX: there's no direct visual read or page count — instead run `~/projects/cv-tailor/venv/bin/python3 ~/projects/cv-tailor/scripts/docx_stats.py <output>.json` for a length proxy (total bullets, and total narrative words across summary/bullets/extra sections), and sanity-check the JSON content record itself for structural issues (empty required fields, a section that ended up empty when the source had content, obviously mismatched data in the wrong field). As a rough guide for the default template's font/margins, comfortably-1-page output tends to land around 550-650 narrative words and under ~16-18 total bullets — apply the same 1-page-default / 2-page-only-if-justified test from the PDF branch above using this proxy in place of an exact page count.
   - Re-render if you find and fix anything, or if you cut further for length. **Cap this at 2 additional cut-and-recheck iterations** (mirroring the review cap in step 5) — if it's still over target after that: if the overage is genuinely justified (5+ years experience, all-relevant content), proceed and note that honestly in step 9; if it isn't justified, cut more aggressively rather than shipping an unjustified 2 pages.
   - Only proceed to step 9 once the output actually looks right and you've made a deliberate decision on length (on target at 1 page, or a full 2 pages justified by experience + relevance).

9. Tell the user where the output file is, briefly summarize what was changed and why (e.g. "led with your data-pipeline experience since the posting emphasizes ETL work; trimmed the unrelated retail bullet"), and include a short QA note from the review step (e.g. "review pass: no unsupported claims found" or "review caught and fixed one bullet that overstated scope"). If the source was a long master CV, mention what was condensed at a high level (e.g. "condensed from 6 roles down to the 3 most relevant, cut ~40 bullets to the ~12 strongest matches for this posting") so the user can see what was cut and why, and can ask for anything back if they disagree with a cut.
